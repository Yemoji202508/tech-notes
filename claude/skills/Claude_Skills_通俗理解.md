# Claude Skills 通俗理解指南

> 基于刘小排文章《Claude Skills 不就是把提示词存个文件夹吗？》的通俗解读

## 💡 记忆口诀

**Prompts**：说了就忘  
**Skills**：写进手册  
**Projects**：建个档案室  
**Subagents**：请个专职员工  
**MCP**：接通外部世界  

---

## 核心观点

**Skills 不是简单地把提示词存成文件夹，而是把"提示词工程"升级为"流程工程"。**

---

## 一、用"开餐厅"理解 Claude 生态的五大组件

> **说明**：这一章用"开餐厅"作为类比工具，帮助你快速理解五大组件的本质。  
> 后面会有完整的实战案例，展示它们如何在真实工作中配合使用。

### 🍳 Skills = 菜谱和烹饪手册

**是什么**：写好的标准操作流程，需要时自动调用

**餐厅例子**：
- 你有一本《川菜烹饪手册》
- 里面写着：宫保鸡丁怎么做、麻婆豆腐的标准配方、火候控制技巧
- 当客人点川菜时，厨师（Claude）会自动翻开这本手册
- 不点川菜时，这本书就放在架子上，不占用厨师的工作台空间

**真实场景**：
```
你创建一个"品牌文案 Skill"：
- 标题必须 15 字以内
- 开头用痛点提问
- 结尾要有行动号召
- 禁用"赋能""生态"等词汇

以后让 Claude 写文案，它会自动套用这些规则
```

---

### 💬 Prompts = 当场的口头指令

**是什么**：你现在、立刻、马上说的话

**餐厅例子**：
- "今天这道菜少放点盐"
- "给 8 号桌的客人加个甜品"
- "帮我算一下今天的营业额"

这些指令**说完就过去了**，明天不会自动记得。

**真实场景**：
```
你对 Claude 说：
"把这段话改得更专业一点"
"用项目符号重新排版"
"重点分析医疗行业的数据"

这些都是一次性指令，下次对话需要重新说
```

---

### 📁 Projects = 餐厅的档案室

**是什么**：一个专门的工作空间，里面存着所有背景资料

**餐厅例子**：
- 你开了一家"川菜主题餐厅"项目
- 档案室里放着：
  - 供应商联系方式
  - 过去 3 个月的销售数据
  - 客户反馈记录
  - 菜单设计稿
- 所有关于"川菜餐厅"的对话都在这个空间里进行
- 厨师（Claude）随时能查阅这些资料

**真实场景**：
```
你创建一个"Q4 产品发布"项目：
- 上传市场调研报告
- 上传竞品分析文档
- 上传产品规格说明
- 设置项目指令："所有分析要站在 B 端客户视角"

在这个项目里的所有对话，Claude 都能看到这些背景资料
```

**关键区别**：
- **Projects** = 知识库（"你需要知道什么"）
- **Skills** = 方法论（"你应该怎么做"）

---

### 👨‍🍳 Subagents = 专职员工

**是什么**：专门负责某项工作的独立助手（仅限 Claude Code/SDK）

**餐厅例子**：
- **主厨**（主 Agent）：统筹全局
- **凉菜师傅**（Subagent 1）：只负责凉菜，只能用切菜刀和冰箱
- **面点师傅**（Subagent 2）：只负责面点，只能用烤箱和面粉
- **收银员**（Subagent 3）：只负责结账，只能看账单，不能进厨房

每个员工有自己的工作台（独立上下文），有自己的工具权限。

**真实场景**：
```
你在 Claude Code 里创建三个 Subagents：

1. code-reviewer（代码审查员）
   - 只有 Read、Grep 权限（只读）
   - 专门检查代码安全问题
   - 不能修改代码

2. test-writer（测试编写员）
   - 有 Read、Write 权限
   - 专门写单元测试
   - 不能改主代码

3. doc-writer（文档编写员）
   - 有 Read、Write 权限
   - 专门写 API 文档
   - 只能操作 docs/ 目录
```

---

### 🔌 MCP = 餐厅的外部供应系统

**是什么**：连接外部数据和工具的"总线"（开放协议）

**餐厅例子**：
- 连接**供应商系统**：自动查看今天有哪些新鲜食材
- 连接**点评网站**：实时看到客户评价
- 连接**库存系统**：知道哪些菜快卖完了
- 连接**天气预报**：根据天气调整菜单

**真实场景**：
```
你给 Claude 接入 MCP：

1. Google Drive MCP
   → Claude 能读取你的云端文档

2. GitHub MCP
   → Claude 能查看代码仓库

3. Slack MCP
   → Claude 能读取团队聊天记录

4. 数据库 MCP
   → Claude 能查询业务数据
```

**关键区别**：
- **MCP** = 连接数据（"能访问什么"）
- **Skills** = 使用数据的方法（"怎么用这些数据"）

### 📊 快速回顾

| 组件 | 餐厅类比 | 一句话 |
|------|----------|--------|
| **Skills** | 菜谱手册 | 写好的标准操作流程 |
| **Prompts** | 口头指令 | 当场临时说的话 |
| **Projects** | 档案室 | 存放背景资料的空间 |
| **Subagents** | 专职员工 | 独立负责某项工作 |
| **MCP** | 供应系统 | 连接外部数据和工具 |

## 二、Skills 的核心技术：渐进披露

### 工作原理

```
第一步：扫描"封面简介"（~100 tokens）
├─ Claude 快速浏览所有 Skills 的元数据
└─ 判断：这个 Skill 跟当前任务有关系吗？

第二步：读取"说明书正文"（<5k tokens）
├─ 确认相关后，加载完整的 SKILL.md
├─ 包含：步骤、流程、注意事项、输出格式
└─ 目的：知道具体怎么做

第三步：加载脚本和文件（按需）
├─ 只有真正需要执行时才加载
└─ 包含：代码、模板、示例文件
```

### 为什么这样设计？

**类比**：就像图书馆的书架
- 你不会一次把所有书都搬到桌上
- 先看书名（元数据）找相关的
- 再翻开目录（说明书）确认内容
- 最后才仔细阅读需要的章节（脚本）

**好处**：
- 可以给 Claude 装很多 Skills
- 不会一开始就把上下文撑爆
- 只在需要时加载需要的内容

## 三、完整实战案例：准备一场商务宴请

> **说明**：前面用"开餐厅"是为了帮助理解五大组件的概念（类比）。  
> 现在我们来看一个真实的工作场景（实战），展示这些组件如何在实际业务中协同工作。

### 场景设定
客户是一家医疗科技公司的 CEO，你要在餐厅谈合作。

### 五大组件如何配合

```
【第一步】创建 Project："医疗客户商务宴请"
上传内容：
- 客户公司的背景资料
- 过往邮件往来
- 医疗行业报告
- 你们公司的产品介绍

项目指令：
"所有沟通要体现专业性，重点突出我们在 AI 辅助诊断方面的优势"

【第二步】连接 MCP 数据源
- Google Calendar MCP：查看客户的时间安排
- 大众点评 MCP：查找合适的餐厅
- 天气 API MCP：确认当天天气

【第三步】加载 Skills
1. 商务礼仪 Skill
   - 自动应用商务场合的沟通规范
   - 座位安排、点菜礼仪、话题禁忌

2. 医疗行业 Skill
   - 使用行业术语和关注点
   - 了解医疗行业的监管要求和痛点

3. 餐厅推荐 Skill
   - 筛选标准：环境安静 + 停车方便 + 人均 500+
   - 优先选择客户公司附近
   - 避开嘈杂的网红餐厅

【第四步】配置 Subagents（Claude Code）
1. research-agent（调研专员）
   - 工具：Read、Web-search
   - 职责：调研客户公司最新动态、融资情况、产品发布
   - 输出：一份客户背景简报

2. schedule-agent（日程协调员）
   - 工具：Calendar、Email
   - 职责：协调双方时间，发送邀请邮件
   - 权限：只能读写日历和邮件

3. presentation-agent（演示准备员）
   - 工具：Read、Write、PowerPoint
   - 职责：准备 5 分钟电梯演讲稿和演示材料
   - 要求：突出差异化优势

【第五步】你的 Prompt（实时指令）
"重点强调我们在 AI 辅助诊断方面的优势"
"餐厅最好在客户公司附近，方便他们"
"准备一份 5 分钟的电梯演讲稿"
"改成周四下午，客户那天有个重要会议"
```

### 背后发生了什么？

```
你说："帮我安排下周三和 XX 医疗的 CEO 见面"

↓

Claude 自动执行：

1. 【Project 上下文加载】
   → 读取客户背景资料
   → 了解过往沟通历史
   → 知道你们公司的产品定位

2. 【MCP 联通数据】
   → 查看双方 Google Calendar
   → 在大众点评搜索合适餐厅
   → 查询下周三的天气

3. 【Skills 启用】
   → 商务礼仪 Skill：规划见面流程
   → 医疗行业 Skill：准备行业话题
   → 餐厅推荐 Skill：按标准筛选餐厅

4. 【Subagents 并行工作】
   → research-agent：查客户公司最新新闻
   → schedule-agent：发送邀请邮件
   → presentation-agent：准备演示材料

5. 【你通过 Prompt 微调】
   你："改成周四下午，客户那天有个重要会议"
   Claude 立即调整方案，重新协调时间
```

### 最终输出

一份完整的商务宴请方案：
- ✅ 时间：周四下午 6:30（双方都有空）
- ✅ 地点：客户公司附近的安静餐厅（已预订）
- ✅ 天气：晴天，适合见面
- ✅ 客户背景：最新融资、产品动态、关键人物
- ✅ 演示材料：5 分钟电梯演讲 + PPT
- ✅ 话题准备：医疗 AI 监管、行业痛点、合作机会

## 四、案例总结：为什么商务宴请案例好？

### 体现了真实的工作场景
- 不是技术 Demo，而是实际业务需求
- 涉及多个系统协同（日历、邮件、点评、天气）
- 需要专业知识（商务礼仪、行业知识）

### 清晰展示了各组件的价值
- **Project**：提供客户背景，不用每次重新介绍
- **MCP**：连接真实数据源，不是手动查询
- **Skills**：自动应用专业规范，不用每次提醒
- **Subagents**：并行处理任务，提高效率
- **Prompts**：灵活调整细节，保持人的控制权

### 容易理解和记忆
- 每个人都能想象商务宴请的场景
- 类比生活经验，降低理解门槛
- 记住这个案例，就理解了整个生态

## 五、关键对比表

| 组件 | 类比 | 持久性 | 包含内容 | 加载时机 | 最适合 |
|------|------|--------|----------|----------|--------|
| **Prompts** | 口头指令 | 单次对话 | 自然语言 | 每次输入 | 临时请求 |
| **Skills** | 菜谱手册 | 跨对话 | 指令+代码+资源 | 动态按需 | 专业能力 |
| **Projects** | 档案室 | 项目内 | 文档+上下文 | 项目内始终 | 背景知识 |
| **Subagents** | 专职员工 | 跨会话 | 完整 Agent 逻辑 | 被调用时 | 专项任务 |
| **MCP** | 供应系统 | 持续连接 | 工具定义 | 始终可用 | 数据访问 |

## 六、什么时候该用什么？

### 用 Skills 的场景

✅ 组织级工作流
- 品牌规范、法务流程、文档模板

✅ 专业领域经验
- Excel 公式、数据分析套路、代码 Review 标准

✅ 个人偏好习惯
- 笔记结构、代码风格、研究方法

**判断标准**：任何你不想一遍遍重新解释的东西

### 用 Prompts 的场景

✅ 一次性请求
- "总结这篇文章"

✅ 对话式调整
- "语气改专业一点"

✅ 即时上下文
- "分析这份数据"

**升级信号**：如果同一个 Prompt 反复敲，就该做成 Skill

### 用 Projects 的场景

✅ 需要长期背景知识
- 某个产品线、某个大客户、某个长期课题

✅ 需要工作隔离
- Q4 产品发布、某场活动运营、某轮融资材料

✅ 团队协作（Team/Enterprise）
- 共享历史对话、共享知识库

### 用 Subagents 的场景（仅 Claude Code/SDK）

✅ 任务专业化
- 代码审查、单元测试生成、安全审计

✅ 上下文拆分
- 主对话保持干净，重活丢给 subagent

✅ 并行处理
- 多个 subagent 同时工作

✅ 工具权限隔离
- 某些 subagent 只读，永远不能写入

### 用 MCP 的场景

✅ 访问外部数据
- Google Drive、Slack、GitHub、数据库

✅ 使用业务工具
- CRM 系统、项目管理平台

✅ 连接开发环境
- 本地文件、IDE、版本控制

✅ 集成自研系统
- 公司内部平台

## 七、作者的核心判断

### Prompt Engineering 只是上半场

- Prompt 解决的是"怎么和模型说话"
- Skills/Projects/Subagents/MCP 解决的是"模型在复杂环境下，怎么长期、稳定、可维护地干活"

### 真正的下半场是"流程工程"

对 AI 产品创业者的启示：

1. **差异化体现在 Skills 设计上**
   - 把领域经验、工作流、组织规范沉淀成 Skills 库
   - 这是护城河，不只是"调了哪个大模型"

2. **产品架构是"Agent 积木组合"**
   - 某个功能 = 若干 Project + Skills + Subagents + MCP 的组合
   - 未来可能有 Skill Store

3. **团队需要新角色**
   - 不只是写 Prompt 的人
   - 而是"Skill Engineer / AI Workflow Architect"
   - 负责把经验变成可复用的 AI 工作流
